Devops之基于Jenkins的CI与CD实战

一、 对Jenkins实现代码的部署和回滚

1.1 实现对测试环境和生产环境代码部署

​ 基于参数控制，基于传递分支，如果master就是生产环境，develop就是测试环

​ 对于虚拟机或者物理机：

​ 1.master=10.0.0.106，10.0.0.107

​ 2.develop=10.0.0.105

​ GROUP1=10.0.0.105 #灰度服务器

​ GROUP2=10.0.0.106 10.0.0.107 #线上服务器

​ GROUP3=10.0.0.105 10.0.0.106 10.0.0.107

​ 对于容器，不需要指定地址

​ 只要再k8s执行代码更新就行

​ 对于公司有CMDB

​ 通过CMDB的API自动获取当前环境当前服务器地址

​ MASTER=$Host

1.2 实现对测试环境和生产环境代码的回滚

基于git reset

​ 需要重新执行代码编译，代码编译时间较长

重新升级

​ 开发修复玩bug执行重新提交代码，也是需要重新执行代码编译，代码编译时间比较长

基于本分代码目录回复，不需要代码编译，回滚时间少

​ 基于已经存在的历史版本目录

​ 1.我现在所处的版本

​ 如果今天升级过的版本 v1 v2 v3 ，最新是v3，

​ 2.我的上一个版本是谁

​ v3

​ 3.将webapps/myapp 软连接重新指向上一个版本

​ 4.重启tomcat

执行shell里的命令一般为执行shell脚本或者python脚本，而脚本的内容则为Jenkins代码的部署流程。

Q：为什么不像以前那样直接把流程写在Jenkins的网页里呢？

A：防止别人有意或无意对Jenkins网页里的脚本进行修改，而在Jenkins服务器里写成脚本我们可以适应gitlab进行脚本的备份，这样可以在shell脚本被修改后可以立即还原。
